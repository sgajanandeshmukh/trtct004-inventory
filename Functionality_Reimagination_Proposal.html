1. Executive Summary

The TRTCT004 Inventory Management System has been successfully migrated from IBM AS/400 RPG to a modern Spring Boot + React stack. While this migration preserves all 70+ legacy business rules, it only scratches the surface of what a modern platform can deliver. This proposal outlines 12 inventory-domain functionality reimaginations plus 2 cross-cutting infrastructure prerequisites that transform inventory management from a transactional record-keeping system into an intelligent, predictive, and connected platform вҖ” all strictly within the scope of inventory management.

2.3 Identified Gaps & Unrealized Potential

GAP-1 Dormant Transaction Table
The INVTRN (inventory transaction log, DB-005) table was created in V1 migration with full schema вҖ” but no service, repository, or controller writes to it. This eliminates all audit capability.
GAP-2 Static Reference Adapters
Category, Supplier, and Warehouse adapters use hardcoded static maps instead of querying the database tables (CTGMST, SUPLMST, WHSMST) that were seeded in V5 migration.
GAP-3 No Inter-Warehouse Visibility
Each item is assigned to a single warehouse. There is no mechanism for warehouse-to-warehouse transfers, consolidated stock views, or optimal sourcing logic.
GAP-4 Report Print Stub
ReportPrintAdapterProxy.spoolReport() is a stub. Legacy INVRPTR (PGM-004) printed physical spool files вҖ” this capability was not re-implemented, leaving reporting incomplete.

3. Reimagination Vision & Principles

3.1 Vision Statement

Transform WorkStream Inventory Management from a transactional record system into an intelligent inventory platform that predicts demand, automates replenishment, enables real-time warehouse operations, and provides actionable business intelligence вҖ” all while preserving the 70+ battle-tested business rules from the legacy system.

3.2 Guiding Principles

#PrincipleDescription
P1Intelligence Over EntryAutomate decisions that were previously manual. The system should recommend, not just record.
P2Real-Time Over BatchReplace legacy batch processing patterns with event-driven, real-time data flows.
P3Predictive Over ReactiveUse historical data and algorithms to anticipate needs instead of responding to shortages after they occur.
P4Connected Over IsolatedBreak silos between inventory, suppliers, warehouses, and payroll systems through APIs and integrations.
P5Mobile Over Desk-BoundBring inventory management to the warehouse floor with mobile-first experiences.
P6Auditable Over OpaqueEvery inventory movement must be traceable, compliant, and explainable.
P7Preserve & ExtendAll 70+ existing business rules (BR-001 through BR-070) remain intact. New capabilities layer on top, never replace validated logic.

4.2 Predictive Reorder & Demand Forecasting

FR-002 Evolve static reorder points into ML-driven demand prediction

Legacy Context: The reorder report (SCR-006) uses a simple threshold comparison: quantityOnHand ≤ reorderPoint (BR-062). Reorder points and quantities are manually maintained per item вҖ” they never adapt to actual demand patterns.

Reimagined Capability:

FeatureDescriptionType
Consumption Velocity TrackingCalculate daily/weekly burn rate per item from transaction history. Replace static reorderPoint with dynamic safety stock levels.Reimagine
Seasonal Demand CurvesIdentify seasonal patterns (quarterly, monthly) and adjust reorder timing proactively. Items ordered 2 weeks before predicted demand spike.New
Lead Time AwarenessFactor supplier lead times into reorder calculations. Trigger reorder when daysOfStockRemaining < supplierLeadTimeDays.New
Economic Order Quantity (EOQ)Calculate optimal order quantity using EOQ formula: √(2DS/H) where D=annual demand, S=order cost, H=holding cost per unit.Reimagine
Auto-Reorder SuggestionsSystem generates purchase order drafts when predicted stockout date falls within lead time window. Requires human approval to execute.New
Forecast Accuracy TrackingCompare predicted vs actual demand over time. Display forecast accuracy percentage to build system trust.New
Algorithm Evolution: ALG-002 (shortage = max(0, reorderPoint − qtyOnHand)) remains as the fallback. The new predictive layer sits above it: predictedShortage = max(0, forecastedDemand(leadTimeDays) − qtyOnHand + safetyStock). If forecasting data is insufficient (<90 days of history), the system gracefully degrades to the original ALG-002 logic.

4.6 ABC Inventory Classification & Analytics

FR-006 Auto-classify inventory by value contribution using Pareto analysis

Legacy Context: All inventory items are treated equally in the legacy system. A $0.50 packing material receives the same reorder attention as a $5,000 electronic component. There is no prioritization mechanism.

Reimagined Capability:

ClassRuleTypical %Management Policy
A вҖ” CriticalTop items contributing to 80% of total inventory value~20% of itemsTight control, frequent cycle counts, low safety stock, demand forecasting enabled
B вҖ” ImportantNext items contributing to 15% of total value~30% of itemsModerate control, quarterly cycle counts, standard reorder process
C вҖ” StandardRemaining items contributing to 5% of value~50% of itemsSimplified control, annual counts, higher safety stock to reduce order frequency

Algorithm вҖ” ALG-004: ABC Classification

1. For each ACTIVE item: annualValue = quantityOnHand Г— unitCost 2. Sort all items by annualValue descending 3. Calculate cumulative percentage of total value 4. Assign class: - A: cumulative% вүӨ 80% - B: 80% < cumulative% вүӨ 95% - C: cumulative% > 95% 5. Store classification on entity (new field: abcClass CHAR(1)) 6. Recalculate nightly via scheduled job

New Screen Elements: ABC class badge on list screen (SCR-002), ABC distribution donut chart on dashboard (FR-001), classification override capability for manual adjustment.

4.12 Lot Tracking, Expiry & Serial Number Management

FR-012 Enable traceability from receipt through consumption

Legacy Context: The AS/400 system tracks only aggregate quantities per item (IMQTOH). It has no concept of lots, batches, expiration dates, or serial numbers. If a recall is issued for a specific manufacturing batch, there is no way to identify affected stock.

Reimagined Capability:

FeatureDescription
Lot/Batch NumbersAssociate received items with lot numbers from suppliers. Track quantity per lot within an item. Enable FIFO (First In, First Out) consumption.
Expiry Date ManagementRecord expiry date per lot. Dashboard widget showing items expiring within 30/60/90 days. Auto-alert for items nearing expiry (FR-007).
Serial Number TrackingFor high-value items (Class A): assign unique serial numbers. Full lifecycle tracing from receipt вҶ’ location вҶ’ issuance вҶ’ return.
Recall ManagementEnter lot number or date range вҶ’ system identifies all affected stock locations, quantities, and status. Supports quarantine status on affected lots.

Database: New table inventory.lotmst (lot master): lot_id, item_id, lot_number, supplier_lot_ref, received_date, expiry_date, quantity, status. New table inventory.serialmst: serial_id, item_id, lot_id, serial_number, current_location, status.

4.13 Inventory Valuation & Costing Methods

FR-013 Support multiple costing methodologies for financial reporting

Legacy Context: The current system stores a single unitCost per item вҖ” effectively a standard/replacement cost. There is no historical cost tracking, no weighted average calculation, and no FIFO/LIFO costing. ALG-001 calculates margin based solely on the current unit cost.

Reimagined Capability:

Costing MethodDescription
Standard CostCurrent approach (preserved): fixed unitCost manually maintained. Variance = actual purchase price − standard cost.
Weighted Average CostRecalculate cost on each receipt: newAvgCost = (existingQty Г— currentCost + receivedQty Г— purchasePrice) / (existingQty + receivedQty)
FIFO CostingTrack cost per lot (FR-012). COGS uses oldest lot cost first. Requires lot-level quantity tracking.
LIFO CostingCOGS uses most recent lot cost first. For tax optimization in specific jurisdictions.

New Capabilities:

  • Inventory Valuation Report: Total value calculated under selected costing method. Compares valuations across methods side-by-side.
  • Margin Re-Calculation: ALG-001 enhanced to use selected costing method instead of static unitCost. Margin becomes dynamic based on actual historical costs.
  • Cost Variance Analysis: Dashboard showing purchase price variance (PPV) trends over time, by supplier, by category.

4.14 Role-Based Access Control Infrastructure

FR-014 Secure the application with authentication and granular permissions

Legacy Context: The AS/400 relied on OS-level security profiles (USRPRF objects) and library-level authority. The migrated application currently has no authentication or authorization вҖ” every endpoint is publicly accessible.

Why It's Infrastructure, Not Inventory: Every web application needs authentication and authorization. These roles, JWT tokens, and SSO integration are not specific to inventory management вҖ” the same security layer would be needed for Payroll, Procurement, or any other module.

Required Capability:

RolePermissionsUse Case
ViewerRead-only access to all screens. Cannot add, edit, or delete items.Auditors, management
OperatorCreate/edit items, submit cycle counts, scan operations. Cannot delete or change status.Warehouse staff
ManagerAll Operator permissions + soft delete, approve cycle count variances, run reports.Warehouse managers
AdminFull access including reference data management, user management, system configuration, bulk operations.IT / system admin

Implementation:

  • Spring Security with JWT tokens (stateless auth, replacing legacy stateful session model)
  • Warehouse-scoped data isolation: Operators see only their assigned warehouse's items
  • Audit log of all login/logout events and permission-denied attempts
  • PII masking on supplier contact fields (SUPLMST) based on role (addresses GAP in DB-003 PII annotations)
  • OAuth2/OIDC integration for enterprise SSO

Why It's Infrastructure, Not Inventory: API documentation (OpenAPI/Swagger) and webhook infrastructure are generic platform capabilities. The webhook engine, API key management, rate limiting, and event bus architecture would serve any module вҖ” not just inventory.
