2.2 Hexagonal Architecture (Ports & Adapters)

Pattern Applied: The legacy RPG programs directly accessed DB2/400 files and external reference tables with hardcoded READ/WRITE operations. The modernized application uses a hexagonal architecture where domain services interact through port interfaces, with concrete adapters handling external concerns.

2.3 State Machine Formalization (STM-001)

Reimagination: The legacy system managed item lifecycle status through scattered RPG conditional logic (IF IMSTAT = 'A' / MOVE 'I' IMSTAT). The modernized system formalizes this into an explicit InventoryItemStateMachine component with a declared transition map, enforced guard conditions, and a dedicated exception type.
TransitionLegacy ImplementationModern Implementation
T-001: ACTIVE → INACTIVEIF IMSTAT='A'; MOVE 'I' IMSTAT; UPDATE ITMMSTstateMachine.transition(entity, INACTIVE, "T-001") with guard + InventoryItemStateTransitionException
T-002: INACTIVE → ACTIVENot in original scopeentity.reactivate() — formalized for future use
T-003: ACTIVE → PENDING_DELETENot in original scopeentity.flagForDeletion() — reserved for approval workflow
PENDING_DELETE → DELETEDPhysical DELETE operationLogical terminal state — no physical delete

6. Validation Reimagination

6.1 Dual-Layer Validation

Key Reimagination: The legacy system performed validation only server-side after the user pressed Enter, displaying a single error message at a time. The modern system implements dual-layer validation: client-side Zod schema validation for instant feedback, plus server-side Spring validator for business rule enforcement. All errors are accumulated simultaneously (BR-048).
LayerTechnologyRules EnforcedUser Experience
Client (Browser)Zod schema + React Hook FormField presence, length limits, numeric ranges, price ≥ cost cross-field checkInstant inline error on blur/submit — no server round-trip
Server (Spring)InventoryItemValidatorBR-036 to BR-048: ID required/unique, name required, category/supplier/warehouse code existence, price ≥ cost422 response with accumulated error codes (INV-001 to INV-007)

7.2 Typed API Client

New A fully typed Axios-based API client (inventoryApi.ts) provides TypeScript interfaces for every request/response, enabling compile-time type safety between frontend and backend contracts. The legacy system had no equivalent — screen buffer I/O was untyped.
